	Parameter
	─────────
	in relation to programming, is a list of one or more requests made by an item to be fufilled when it is used.

	Types of Parameter
	──────────────────
	1. Value expression parameter (Think more about the name)
	2. Type parameter

	Type parameter
	──────────────
	is the type of parameter that is used for requesting types.

	Argument
	────────
	in relation to programming, is a list of one or more answers to the request of a parameter when it is used. Since there are only two types
	type of parameters, there are also only two types of arguments namely:

	1. Type arguments
	2. Value expression arguments

	Argument Entry
	──────────────
	is simply the entry process of one or more arguments to their parameters. Same applies for argument entry types, namely:

	1. Expression argument entry
	2. Type argument entry : If the arguments are zero the angle braces can be omitted

	Argument Special Entry Types
	────────────────────────────
	1. Explicit default argument entry [Should be renamed to Default argument entry]
	2. Implicit default argument entry [Should be a subtopic under default argument entry called implicit default argument]
	3. Parameter-argument entry
	4. Value argument list generator argument entry

	Explicit Default Argument Entry
	───────────────────────────────
	parameters can specify arguments for themselves to use, these arguments are called default arguments. To then explicitly use a
	default argument, the placeholder identifier [_] is used. The use of the placeholder identifier[_] is known as explicit default
	argument entry. Explicit default argument entry is a feature of both value-expression and type argument entry.

	```(nc-lang)
	fn fcn(type si32 _ = 45){} // fcn(_)

	struct lose<type _ = si32> // type lose<_> x()
	```
	For a value-expression parameter that is based on a type parameter, to specify a default argument for itself, the type parameter
	must first have a specified default argument.

	```(nc-lang)
	fn fcn<type t= si4>(type t x = 90'si4){}
	```

	Implicit Default Argument Entry
	───────────────────────────────
	With implicit default argument entry, the placeholder identifer[_] need not be used only when they are the last arguments to be
	entered. Implicit default argument entry is also a feature of both value-expression and type argument entry.

	```(nc-lang)
	fn fcn<type t = si4>(){} // fcn<>() or fcn()
	fn some(type ui4 x = 90){} // fcn()
	```

	Parameter-Argument Entry
	────────────────────────
	This type of argument entry simply provides a way to associate arguments with their parameter identifers. Parameter-argument
	entry is only a feature of value-expression argument entry. Parameter-argument entry is the only entry that can be combined
	with default argument entry.

	Syntax
	──────
	'.'<identifier> '=' <expression>

	Parameter-Argument entry is not valid for variadic value-expression parameters and [_] parameters.

	Types of Parameter-Argument Entry
	─────────────────────────────────
	1. Ordered parameter-argument entry
	2. Unordered parameter-argument entry

	Ordered Parameter-Argument Entry
	────────────────────────────────
	Is when parameter-arguments are entered orderedly. Because they are ordered, they can be mixed with non parameter-arguments.

	Unordered Parameter-Argument Entry
	──────────────────────────────────
	Is when parameter-arguments are entered unorderedly. The language allows this because it can be solved easily by requiring that
	from the starting point of the unordered parameter-argument entry, only parameter-arguments are allowed from then on.

	Type Argument Inference
	────────────────────────
	Type arguments can be inferred by the compiler in the certain scenarios and when successfully inferred by the compiler behaves as
	if the type parameter had been provided a default type argument and therefore is subject to default argument entry semantics. The
	following are the defined scenarios:

	When Used As A Type In A Value Parameter
	─────────────────────────────────────────
	When the type parameter of a type argument is used as a type in a value parameter, the type argument can be inferred from the value
	argument entered. This true for any construct that uses both type and value arguments at the same time, examples of which are:
	- Functions with type parameters
	- Function marco with type parameters
	- Functions in a type scope where that type uses type parameters or to be concise functions in a type constructor's scope
	- Object constructors whoose types are type constructors

	```(nc-lang)
	fn fcn[type T](type T _) {}

	:fcn[_](45)
	/# or
	:fcn(45)

	struct data[type T]
	scope @data[type T] {
		fn show(type T _) {}
	}

	/# @data:show(3.4)
	```

	When Used As The Give Type Of A Function or Function Marco
	──────────────────────────────────────────────────────────
	When the type parameter of a type argument is used as the give type of a function or function marco, the type argument can be
	inferred from the typed context the function or function marco when called appears in. A failed inference attempt occurs when it
	is used in an untyped context.

	```(nc-lang)
	fn call[type T]() type T { ... }

	type ui4 _ = call()
	/# or
	infer _ = call() + "Daniel"

	/# Example of use in untyped context

	fn fcn[type T](type T _) {}

	fcn(call()) /# Error
	call().add() /# Error
	infer _ = call()
	```

	Something Like The Following Is Not Possible In Nc-lang But Is possible In Rust
	───────────────────────────────────────────────────────────────────────────────
	```(nc-lang)
	type arrayList[ui2] mut inner_vec = @arrayList:with_capacity(3)
	```

	```(rust)
	let mut inner_vec: Vec<u16> = Vec::with_capacity(3)
	```

	From the above semantics, you can see that inference is selective and defined in it's use. There is need to differentiate
	between a type created from a type constructor that uses inference semantics and actual type constructor used in type parameter
	and arguments. A type constructor can only exist in type parameters and arguments, any rendition outside them are simply type
	created from type constructor following inference rules.

	Comptime Value Argument Inference
	─────────────────────────────────
	The use of a comptime value argument in parameter or give type successfully infers the comptime value argument.
	```(nc-lang)
	struct unknown!(type ui4 _)

	fn done!(type ui size)(type [size]ui8 _) {}
	fn back!(type ui4 num)(type unknown!(num) _) {}

	infer _ = done([3, 4, 5, 6])
	infer _ = back(type unknown!(23))
	```

	: Function expression inference
	{
		Given that function expressions are runtime constructs, they are devoid of comptime constructs like LCI attributes, type
		parameters and comptime parameters. But they support something that function declarations do not have, they allow the use of
		`infer` to support inference or one-time inference when juxtapose with function declarations. Using `infer` for a value parameter
		forces the compiler to try to infer the parameter from subsequent typed contexts, like function types and object-function call.

		```(nc-lang)
		type fn(si4)! x = fn(infer a){!print(a)}

		fn const[type T](type T fcn) apply T impls[T] fn(ui4)ui8 {
			(fcn as fn)(45)
		}

		const(fn(infer a){a as ui8})
		```

		Note that this inference follows the same semantic that give type inference uses, which is supported by both function and function
		expressions.

	{
		Untyped number literal semantics are now for both primitive integer types and semantic ranged integer types

		Unlike text literals tokens, number literal tokens do not have a default assigned literal type when none is specified by the
		user. A number literal token that is devoid of a literal type is called an untyped number literal. The literal type of an
		untyped number literal token is inferred by the compiler in situations where it is possible to be inferred and given default
		literal types - which are `ui4` for integer number literal types and `r4` for real number literal token in situations where
		it cannot be inferred. Note that this inference is only possible for primitive number literal types because they are
		ubiquitous to the language and inference behaviour like this is not favourable to an open set of appilicable types, a closed
		set of well known types increases the favourability of such an anti feature to a static and strong typed langauge.

		As nc-lang is against any form of implicit conversions it is important to make the distinction that the following implicit
		inferrence is different than a language that assigns a default literal type to a number literal token and proceeds to perform
		implicit conversions if the types do not match.

		```(nc-lang)
		/# Example 1 of inference

		fn fcn(infer x) {x+1}
		fn cost(type si4 x) {x + 2}

		fn main {
			infer x = 5
			infer y = x + 5
			infer z = fcn(x)
			infer a = cost(y) /# First typed use

			!printf("%{x}=, %{y}=, %{z}= and %{a}=")
		}

		/# Example 2 of inference
		/# In the case of collection expressions, the compiler takes advantage of the fact that they are homogeneous

		fn use {
			type [1, 2]ui4 x1 = [2, 3, 4, 5] /# First typed use
			infer x2 = [2, 3, 4'ui4, 5] /# Inferred from typed element
			infer y = [3, 4, 5, 6]
			infer x = y.!nth(2) + 3'ui4 /# First typed use
			type arrayList[ui4] y2 = [2, 3, 4, 5, 6] /# Also inferred
		}
		```

{
	ZSTs are compile time high-level constructs that are only present within the type system but disappear at run time.

	{

		union type creator	: cannot create ZSTs because of it's semantics does not support it. But It can contain ZSTs.
		set type creator	: cannot create ZSTs because of it's semantics does not support it. Requires the user of the type to specify atleast of value of the type
		unique type creator	: can create ZSTs if the type used in creation is a ZST.
	}
	2. Handling and-fields(the proper name for an object in a struct type creator in nc-lang) that are zero size

: Failure handling types
{
	When an operation fails the context it is used in determines if it is errorneous or defined behaviour

	- Error management type
	{
		Errors in nc are treated just as values but with special language syntax to make working with them easier.
		A type that uses the special syntax of errors is called an error management type, the following are constraints of an error
		management type:
		- The type-creator must be a union type creator of binary or-fields
		- The language treats the first or-field as the expected value while the second or-field as the error value.
		- The first or-field cannot be an error management type
		- The second or-field cannot be a ZST
		- LCI createsErrorManagementType attribute is used to make type-creators error handlers
		- ok or-field can be a zero sized type
		- err or-field cannot be a zero sized type
	}
	- Optional management model
	{
		The special syntax for error management are also used for optional management. The following are constraints of an optional
		management type:
		- The type creator must be union type creator of binary or-fields
		- The second or-field must be a ZST
		- The first or-field is not allowed to be a ZST
	}

: Operator overloading
{
	- To ensure coherence of where operator overloads are defined, all operators must be overloaded as type member items.
		Meaning an operator overload must be a member of a type. Note they can still be inside contracts since they are directly
		implemented for types in their (type)scopes, but they must be member items of the type.
	- All operators are to be overloaded in a marco
	- The LCI `'op`<operator> is used in operator overloading
	{
		Example:
		```
		struct point {type si4 (x, y)}

		scope @point for
		marco 'op+(obj (self, other): point) {
			-> obj: point(self.(x, y) + other.(x, y))
		}
		end scope @point
		```
	}
	- Operators with limitations on overloading
	{
		- Dereferencing comes with the limitation that the subject of the dereference must be a reference
		- An operator must be overloaded in the type scope of the first parameter it uses.
		- There are overloadable operators and non-overloadable operators
		- All overloadable operators must always ask for immutable parameters
		- Relational operators overloads, which fall into overloadable operators, must always give a boolean expression
	}
	- Intrinsic operators cannot be overloaded
	{
		- The error propagation cannot be overloaded because it is an intrinsic operator, but it can be used to specify an operation to execute before error propagation happens
		{
			```
			scope @result<ui4, error> {
				marco_fn #misc:operator?(type result<ui4, error> self) ! {
					self =>(err){log(err)}
				}
			}
			```
		}
	}
	- Arithmetic operators taking the same types as inputs must give the same type as output
	{
		Meaning something like the below is not allowed:
		```(text)
		type(ui4) + type(ui4) = type(ui8)
		```
	}

	- Block expressions
	{
		In the context of expressions, blocks are known as block expressions with an additional syntax for labelling and semantics that
		enable them to be used as expressions.

		- Additional syntax for labelling
		```(nc-lang)
		{.<identifier>}
		```
		The identifier is called a label and it's namespace is limited to the parent block it is defined in. This labels are used by the
		break and toLoopEnd jump statements which would be discussed further down.

		Block Expression semantics defines that if the last item is a value expression in a block expression, then it is the give value of
		that block expression. Meaning that the last item must be a value expression for it to be the give value of that block, if the
		last value is not a value expression or expression at all, the block expression evaluates to a non-value expression.

		- Example
		```(nc-lang)
		type si4 b
		b := {infer x = 45; x}
		debug:!dump(b)
		```

		ITEMS THAT MAKE USE OF BLOCK EXPRESSIONS
		- Conditional expression: all conditional expression make use of block expressions as the body of the branches and are therefore
		subject to the same semantic as block expressions.
		- Loop statements: all loop statements make use of block expressions as the body of the loop but are not allowed to make use of
		the block expression semantic, only the labelling syntax for break and toLoopEnd use cases.
		- Function declarations: function declarations make use of block expressions as the body of the function and are subject to the
		same semantics as block expressions in addition with it's function defined semantics for give jump statements.
	}
	- Under block expression semantics
	{
		If the last item in a block expression or conditional expression is an expression, then it automatically
		becomes the give value of the (value)expression, the expression cannot be a non-value expression.
		Function blocks do not follow this semantics though, so define semantics that defines the sanity of the
		above.
		```(nc-lang)
		fn foo() {
			-> 34
			-> {
				if true {34}else {23}
			}
			-> if true {2}else {0}
		}
		```
	}
	- Conditional expression branch coverage
	{
		Branch coverage for conditional expression branches when in context of giving a value expression to an expression or function and delay
		initialization
		#at:enforce_branch_coverage

		CONDITIONAL EXPRESSION LEAK
		A conditional expression leak occurs when a condtional expression is written in a way that does not account for all the possible
		conditional branches available for a particular value set. To prevent against condtional expression leak a constraint called
		branch coverage is employed, it is name such because it ensures that a condtional expresion achieves 100% coverage of all possible
		condtional branches for a particular value set.

		CONSTRAINT RULES
		- For the case of or-field query expression which is a condtional expression, either all the or-field condtional branches are
		present or an else branch is present.
		- For the case of a match-with conditional expression when operating on the values from a valueDef type creator, first of the
		values must be comptime and second either all the conditional branches are present or an else branch is present.
		- For the case of match-with operating on any other value set and if conditional expression, an else branch must be present.

		Branch coverage is enforced on condtional expressions in value expression context by default, this is because conditional
		expression leak is not semantically not feasible in a value expression context due to the fact that value expressions have the
		semantic of always evaluating to a value when used. But they are not enforced on conditional expressions in non value
		expression context by default, to enable the constraint qualify the conditional expression with the `enforce_branch_coverage`
		LCI attribute, the only conditional expression when it is appilicable in every context is that of a match-with condtional
		expression operating on valueDef type creator values.

		`#at:enforce_branch_coverage`

		When a delay initialization occurs in a conditional expression branch, that is a conditional expression present in a non-value
		expression context, the langauge requires the user/programmer to qualify the conditonal expression with the `enforce_branch_coverage`
		LCI attribute, except for the compusolry case of a match-with condtional expression operating on values from a valueDef type
		creator
	}
	- Value expression semantic exception for conditional expressions used in a value expression context
	{
		There is a value expression semantic exception for conditional expressions used in a value expression context which states
		that the last item in the block which represents the give value can be subsituted with a give jump statement or function or
		function marco containing the exit function.

	Types

	There are two properties that govern the behaviour of identifier-items namely
	1. Scope
	2. Namespace

	Namespace
	─────────
	refers to the naming space of identifer-items where the language requires each identifier of an identifier-item in a namespace be unique
	so as not to have an identifier conflict. It basically means that in any particular identifer-item namespace, an identifier must appear only
	once. Identifier-items in distinct namespaces can never have identifier conflict with one another while identifer-items with the same
	namespace have more room for identifier conflict since it now has to make sure that there is no identifer conflict between itself and the
	other identifer-items it shares namespace with. The language does have an exception for idenifier conflict which would be discussed later.

	Types
	─────
	- Contract and impl namespace
	- Type creators namespace
	- scope namespace
	- Function and object namespace

	Techinal Pharse
	───────────────
	If multiple identifier-items are in the same namespace, it is said that they share the same namespace. Keyword here is 'share'
	because the same identifier-item is does not share namespace with itself, it dosen't make sense to say that.

	SCOPE
	─────
	refers to the area where identifiers are declared and made accessible. All identifier-items have their own scope which would be
	discussed briefly.

	Examples
	────────
	- Identifier-item scope
	- Expression block scope
	- Global or module scope

	Scope and Namespace
	───────────────────
	With the definition of both scope and namespace, it is further defined that identifier-items in the same namespace and scope must
	have unique names for their identifers.
	Meaning something like this:
	```

	scope sameIdentifier =
	end

	end
	```
	Is possible because they are not in the same scope even though the are in the same namespace.

	2. Known scope

	Overloading // Change to object overloading and find all the question blocks in the other file that is related
	───────────
	As mentioned in the definition of namespace, the language has an exception for identifier conflict which is called overloading.
	Overloading is a language feature that allows identifier conflict for both the <function_decl> and <object_decl> identifier-items
	only when the data type of each identifer is different. This is done because the language has mechanisms for resolving each
	identifier.

	Term
	────
	- overload: refers to the individual identifer of an identifier-item involved in the idenifier conflict

	Overload Resolution
	───────────────────
	Is the solution that empowers the overloading language feature. Overload resolution is the process of resolving an overload to it's
	identifier-item. There are two types of overload resolution namely

	1. Using the language overload resolution function [to]
	2. Using the call operator on function overloads

	Using the language overload resolution function
	───────────────────────────────────────────────
	[to<>()] simply accepts the data type you wish to resolve to and the overload. It is the most unambiguous form of overload resolution.

	```
	fn fcn() type si8{} ``to<fn()si8>(fcn)
	fn fcn() type si90{} ``to<fn()si90>(fcn)

	type si89 fcn() ``to<si8>(fcn)
	```

	Using the call operator on function overloads
	─────────────────────────────────────────────
	This method is limited - in as it is only avialable to function overloads and then only works for certain function overloads - and ambiguity
	prone.
	```
	fn fcn(type si34 _){} ``fcn(234`si34)
	fn fcn(type d32fp _){} ``fcn(23.0)
	```

	But it can't work for function overloads with the same parameter type and number but different give type.
	```
	fn voice() type sound{} ``fcn() ??
	fn voice() type image{} ``fcn() ??
	```

	It is also ambiguous when it comes to certain function overloads with type parameters, this is because argument inference entry is a feature
	of type argument entry.
	```
	fn fcn<type t, type t2>(type t2 _){} ``fcn<si32>(45`si9) without the previous overload, this function would have been called
	fn fcn<type t>(type t _){} ``fcn<si32>(45`si9)
	```
	The language resolves this by resolving to the best match.

	If it the call operator method is not preferred, you can always use the function [to] method.
	```
	fn fcn<type t, type t2>(type t2 _){} ``to<fn(si9)>(fcn<si32, si9>)(34`si9)
	fn fcn<type t>(type t _){} ``fcn<si32>(45`si9)
	```

	Note: the function [to] overload resolution method is not possible for constructors, only the call operator method, that means constructors
	are limited in function type specification

	Overload Resolution Conflict With Internal Scope Access
	───────────────────────────────────────────────────────
	For the <function_decl> and <object_decl> identifer-items, the overload resolution methods would to be an alternative for scope access during
	internal scope access when identifier conflict is encountered, but this is not allowed by the language which forces the use of scope access
	regardless of the alternative.

	Take this:
	```
	scope some
	{
		fn fcn{}
		scope son
		{
			fn fcn()type si9{give type si9}

			fn dac
			{
				give to<fn()!>(fcn) ``Scope access is expected here, although function [to] can do this, scope access is still preferred
			}
		}
	}
	```

	But since the expression block item has an anonymous scope, thereby making scope access impossible, this alternative is allowed to be used
	for scope access during internal scope access. Because of this, the overload rule is appiled and thus preventing items <function_decl> and
	<object_decl> who both share the same namespace, from identifier conflict across scope.
	```
	{
		fn fcn{}
		{
			fn fcn{give 56}
			infer _ = to<fn()!>()
		}
	}
	```

: Lifetime tracking implementation (Correct the assertion that comptime friendly category 2 allocating MAR access is always known at comptime)
{
	MEMORY SEGEMENTS
	The lifetime of an object is dependent on the memory segement the object is defined in.

	- stack memory segment 	: gives language defined temporary lifetime
	- heap memory segment 	: gives user defined temporary lifetime
	- static memory segment	: gives langauge defined permament lifetime

	Given the above, every concrete value is given a lifetime number. Non allocating references manage two lifetime numbers, the
	lifetime number of the references themselves and the lifetime number of the value they reference. Lifetime numbers are assigned
	increasingly from top down, meaning an object that lives longer than another object would always have a lesser lifetime number.

	GENERAL RULE FOR NON ALLOCATING REFERENCES
	Non-allocating references must bind to objects with an equal or lesser lifetime number.

	WRITTEN MATHEMATICALLY
	Given two objects [m] of reference type and [n] of non-reference type, their lifetime numbers [lₘ] and [lₙ], if [lₘ] gt [lₙ]
	then [m] cannot reference [n].

	LIFETIME TRACKING OF REFERENCED OBJECTS ALLOCATED ON THE STACK AND STATIC MEMORY SEGEMENTS
	The general rule is sufficient for object

	LIFETIME TRACKING OF REFERENCED OBJECTS ALLOCATED ON THE HEAP
	Due to the lifetime of heap allocated objects being left to the control of the user, the general rule alone is insufficient in
	preventing lifetime errors, so the following rule is added to manage lifetime tracking of referenced objects allocated on the
	heap.

	PREQUISTES FOR THE RULE
	- The lifetime number assignment for the objects behind an allocating MAR works differently, they are assigned based on the
	  current owner of the objects behind an allocating MAR that is the allocating MAR itself. Assigning allocation wise is
	  inefficient but the an allocation is still to signal to the compiler to act on the constraint.
	- An allocating MAR is the only primitive type that has move value semantics.
	- The language is aware when a heap allocation and deallocation happens.
	- There are two types of move operation for allocating MARs
	  1. Moving to be used
	  2. Moving to be deallocated
	- An allocating MAR must be moved before the end of scope it was declared in.
	- Before a move operation of allocating MARs, the lhs operand must be moved(moved to be used or deallocated),
	- Before an allocating MAR is moved to be deallocated, the object(s) it references must be dropped if the object's type
	  implements the drop contract.
	- The language assigns a lifetime number to an object behind a category 1 allocating MAR.
	- The language assigns the same lifetime number to all the objects behind a category 2 allocating MAR.
	- Objects beind an allocating MAR are given new lifetime numbers on possible allocation.

	ADDITIONAL RULE FOR OBJECTS ALLOCATED IN THE HEAP SEGEMENT
	While following the general rule, a non-allocating-reference that continues to reference an object behind an allocating MAR when
	the allocating MAR could be possibly moved must have a non-overlapping lifetime that is shorter than the object(s) behind the
	allocating MAR unless the non-allocating-reference is part of the object(s) behind the allocating MAR.
	It could also be worded like this:
	"Must have a lifetime that does not overlap with the object(s) behind the allocating MAR and a lifetime number that is greater
	than that of the object(s) behind the allocating MAR".

	- Exact and generic view of the individual objects behind a category 2 allocating MAR (Under comptime)

		reference assignment lifetime relationships are needed or the rigid approach taken by the compiler is used.

{
	- `at:comptime`
	{
		- A non-allocating reference to a comptime value must also be comptime
		- Declaration
		{
			- Function declarations
				Comptime qualifications means the function is now a compile time construct and would run at compile time.
				- Entry function for all nc-lang programs(the main function)
				The main function is the only function allowed to be qualified with the run attribute and when qualified runs the
				entire program at comptime mimicking an interpreted language.

			- Function marco declarations
				Comptime qualifications means the function marco is now a compile time construct and would run at compile time.

			- Type creator declarations
				Comptime qualifications on a type creator means the type must always be comptime qualified and therefore purely
				a compile time construct.

			- Object declarations
				- Value expression parameter
				{
					- Function declarations
						Because functions are runtime constructs, there is need to show the cost of the monomorphisation that would
						occur with the use of comptime parameters and using the comptime attribute on value parameters does not show
						that, so they are not supported, but comptime value parameter entry shows the exactly the cost of
						monomorphisation and should be used instead.

					- Function marco declarations
						Because function marcos are compile time constructs, they fully support comptime value-expression parameters.
				}
				- Local and global objects
					Fully supports comptime.
				- Fields
					Does not support comptime
		}
		- Values that are naturally comptime
		{
			- Primitive literals
			- Value def values
		}
	}
	- `#at:pseudo_comptime` (Disallow pseudo_comptime for parameters in functions)
	{
		- A non-allocating reference to a pseudo comptime value must also be pseudo comptime
		- Declaration
		{
			- Function declarations
				Pseudo comptime qualifications means the function is now partly a comptime construct and would only run at comptime
				when all it's dependencies are comptime.

			- Function marco declarations
				Same with function declarations

			- Type creator declarations
				Does not support pseudo comptime

			- Object declarations
				- Note other pseudo comptime arguments are considered valid arguments for pseudo comptime objects, when used they
				  form a dependency chain.
				{
					```(nc-lang)
					fn fcn1(#at:pseudo_comptime type ui8 x) {}

					fn fcn2(#at:pseudo_comptime type ui8 x) {
						infer _ = fcn1(34) // runs at comptime
						infer _ = fcn1(x) // depends on x which is another pseudo_comptime
					}
					```
				}
				- Value expression parameter
				{
					- Function declarations
						Not allowed for function declarations.

					- Function marco declarations
						Because function marcos are compile time constructs, they fully support pseudo comptime value-expression
						parameters.
				}
				- Local and global objects
				{
					Fully supports comptime.
				}
				- Fields
				{
					struct and union type creator can have comptime fields that would disappear during runtime. Passing them around
					requires for the value-expression parameter to be comptime though even not all the fields are comptime, which
					essentially means they can only use function marcos and comptime functions.
				}
		}

		- Mutable from comptime values accept only from-comptime and comptime values
		- Object declarations
			- Value expression parameter
				General rule
			- Local and global objects
				General rule.
			- Fields

: Grouping syntax and semantics
{
	Groupings
	─────────
	Is a language feature that encapsulates multiple items and allows them to be treated as a single item.
	It's main inclusion to the language is to eliminate unnecessary typing that sometimes leads to code
	verbosity and to add a more high-level touch to the language.
	
	Semantics
	───────────
	Grouping usually consists of two parts, a left and right actor which consitis of participants, A participant
	can either be a single entity or a grouping. Groupings operate by the participants of the left actor acting
	on the participants of the right actor. This type of grouping is called a binary grouping.

	Example:
	```(random)
	[1]
	(x, y) ∘ (a, b)
	(x, y) = left actor
	(a, b) = right actor

	(x, y) => x and y are participants
	(a, b) => a and b are participants

	(x ∘ a, y ∘ b)

	[2]
	(x, y) ∘ (a, (b, c))
	(x, y) 		= left actor
	(a, (b, c)) = right actor

	(x, y) 		=> x and y are participants
	(a, (b, c)) => a and (b, c) are participants
	```

	A grouping can also consist of one part, that is a single actor, which just basically expands by itself. This type of grouping
	is called a lonely grouping. A binary grouping essentially expands to a lonely grouping which is then expanded to normal code
	if it is semantically defined.

	Summary
	───────
	The two types of groupings are:
	- Binary grouping
	- Lonely grouping

	Grouping Levels
	───────────────
	Levels of groupings essentially means the nesting levels of a grouping, just like how blocks can be nested
	example:
	```(random)
	(p₁, (p₁, p₂)₂)
	└───┼──┬─┼┘
		│  1 │
		└──┬─┘
		   2

	(p₁, (p₁, (p₁, (p₁, p₂)₂)₂)₂, p₃)
	└┬───┼────┼────┼──────┼─┼─┼────┘
	 1   │    │    │      │││
		 └┬───┼────┼──────┼┼┘
		  2   │    │      ││
			  └┬───┼──────┼┘
			   3   │      │
				   └┬─────┘
				   	4
	```

	Grouping Amount
	───────────────
	The amount of a grouping refers to the number of participants an actor has.
	```(random)
	(x, b) ∘ b								-- Left actor grouping amount = (1) and Right actor grouping amount = (1)
	(x, b) ∘ (r, m) 						-- Left actor grouping amount = (2) and Right actor grouping amount = (2)
	(x, b) ∘ (r, m, n)						-- Left actor grouping amount = (2) and Right actor grouping amount = (3)
	(x, b) ∘ ((e, b), n)					-- Left actor grouping amount = (2) and Right actor grouping amount = (2)
	(x, (y, (z, (c, g)))) ∘ (a, (b, c), o)	-- Left actor grouping amount = (2) and Right actor grouping amount = (3)
	(x, (y, (z, (c, g)))) ∘ (a, (b, c))		-- Left actor grouping amount = (2) and Right actor grouping amount = (2)
	```

	Grouping Rule Of Use
	────────────────────
	1. User written lonely groupings do not support grouping levels.
	2. The language permits only an expression [and object declaration] to have mutiple levels of a grouping.
	3. The resulting expansion must result in valid ungrouped nc-lang code
	4. Each actor in a binary grouing must have the same grouping amount which ensures full coverage between both actors. Example:
	```(random)
	(b, a, c) ∘ v ✔
	(b, a, c) ∘ (v, g, n) ✔
	{
		🟢 b ∘ v
		🟢 a ∘ g
		🟢 c ∘ n
	}
	(b, a, c) ∘ (f, j) ⤬
	{
		🟢 b ∘ f
		🟢 a ∘ j
		🔴 c ∘ ?
	}
	(x, (y, (z, (c, g)))) ∘ (a, (b, c)) ✔
	{
		🟢 x ∘ a
		[] (y, (z, (c, g))) ∘ (b, c)
		{
			🟢 y ∘ b
			[] (z, (c, g)) ∘ c
			{
				🟢 (z ∘ c, (c ∘ c, g ∘ c))
			}
		}
	}
	(x, (y, (z, (c, g)))) ∘ (a, (b, c), o) ⤬
	{
		🟢 x ∘ a
		[] (y, (z, (c, g))) ∘ (b, c)
		{
			🟢 y ∘ b
			[] (z, (c, g)) ∘ c
			{
				(z ∘ c, (c ∘ c, g ∘ c))
			}
		}
		🔴 o ∘ ?
	}
	(x, (y, (z, (c, g)))) ∘ (a, (b, c, m)) ⤬
	{
		🟢 x ∘ a
		[] (y, (z, (c, g))) ∘ (b, c, m)
		{
			🟢 y ∘ b
			[] (z, (c, g)) ∘ c
			{
				(z ∘ c, (c ∘ c, g ∘ c))
			}
			🔴 m ∘ ? ``dosen't offer full coverage
		}
	}
	```

	- Expressions
	{
		- Expression semantics of precedence and associativity of operators and their operands are still maintained when used in a
		  grouping.
		- An actor of a grouping does not excercise precedence, the precedence which is an expression semantic feature applies only
		  after the binary grouping becomes a single grouping.
		- The above point is there because the precedence operator shares the same brace with grouping syntax which might lead to
		  confusion.
		- Given as expression evaluate to a result, when used on groupings they evaluated and kept as a grouping of the same grouping
		  amount and level.
		- Groupings are generally not allowed to be used in primary expressions because they are meant to represent fundamental
		  expression units within the language, but there are exceptions, like it's use in scope access which would be discussed below.

		```(nc-lang)
		/# Reasons why expression grouping is kept as a group
		{
			((c, v), p) + (k, l) + ((f, h), p)
			/# ((c+k, v+k), p+l) + ((f, h), p) 		-- ((c, v), p) + (k, l) were evaluated and kept as a group
			/# ((c+k+f, v+k+h), p+l+p) 				-- final evaluation

			/# If they were not 
			/# ((c, v), p) + (k, l) + ((f, h), p)
			/# (c+k, v+k, p+l) + ((f, h), p) 		-- ((c, v), p) + (k, l) were evaluated and not kept as a group, there it now violates the 4th rule in the grouping rule of use

		}

		/# Expression semantics are still very much maintained
		&(a, b, c).fcn()					/# -> (&a.fcn(), &b.fcn(), &c.fcn()) -- An example of an actor being deviod of precedence semantics
											/# If it had precedence semantics, it would have evaluated to this : ((&a).fcn(), (&b).fcn(), (&c).fcn())
		(&(a, b, c)).fcn()					/# -> ((&a).fcn(), (&b).fcn(), (&c).fcn())
		(a, b, c).(f, g, l)()          		/# -> (a.f(), b.g(), c.l())
		(a, b, c).(f(), g(), l())           /# -> (a.f(), b.g(), c.l())
		(a, b).(f, g)()*                    /# -> (a.f()*, b.g()*)
		(a, b, c)* += (n, m, c)             /# -> (a* += n, b* += m, c* += c)
		-(a, b)                             /# -> (-a, -b)
		(a, c) * (b, x) + (e, g)            /# -> (a * b + e, c * x + g)
		((c, v), p) =÷ (k, l)               /# -> ((c =÷ k, v =÷ k), p =/ l)
		(x, (y, (z, v))) - (a, (b, c))      /# -> (x - a, (y - b, (z - c, v - c)))
		(x, (y, (z, (c, g)))) + (a, (b, c)) /# -> (x + a, (y + b, (z + c, (c + c, g + c))))

		/# Only postfix unary operators cannot participate in grouping
		/# The call operator cannot participate because it shares the same syntax with the precedence operator and grouping syntax so it is not feasible
		/# x.fcn((2, 4), (4, 5))
		/# The or-field query operator cannot participate because it does not make sense for it's semantic
		/# (x, y) => (some as _ {}or none as _ {}, some as _ {}or none as _ {})
		/# The below is preferrable
		/# (x => some as _ {}or none as _ {}, y => some as _ {}or none as _ {})

		/# Example for swapping objects using an intermediate object as a temporary storage. The two objects to swap are (a, b) and
		/# the temporary object that stores the intermediate object value is (c)
		(c, a, b) = (a, b, c)               /# -> (c = a, a = b, b = c)

		```

		- Context identifier [_] defined use for grouping in expressions
		```(nc-lang)
		/# When used in expressions, it leaves the participant is the other actor unevaluated
		(a, _, b) + (x, y, c) /# -> (a + x, y, b + c)
		/# This is not particularly useful, but in cases where an argument entry is needed for evaluation before passing

		fn pass_something(type (si4, r4, si4) _){}

		fn main()
		{
			infer (a, b, c) = (34'si4, type r4, 56'si4)
			pass_something((a, b, c) * (a, _, c))
		}

		/# Or to leave an object uninitialized which would be shown in the grouing combinations section
		```

		- Lonely groupings are not allowed for expressions
		{
			```(nc-lang)
			fcn(x, (y, z))              /# ERROR: No participant
			fcn((a, (b, (c, (e, f)))))  /# ERROR: No participant
			```
		}
		- The only primary expressions grouping can be applied to are the scope access primary expression and fn identifier entry. All
		  other primary expressions do not support grouping.
		{
			```(nc-lang)
			x:(c, b, n)
			@si4:(c, b, m)
			fn:(dac, fcn)
			y:fn:(dac, fcn)
			z:(fn:(dac, danger)) /# This is dumb but valid
			```
			It is important to note that scope access is not intrinsically an expression(primary expression) which is why they are
			part of import items and type declaration items. Any where that scope items need to be accessed, scoped items would be
			present.
		}

	- Runtime delay initialization support for static arrays (modify later with the new static array syntax)
	{
		Although value expression argument list generator entry makes comptime generation of a list of value expressions possible, there
		is still an obvious comptime and binary cost attached to the feature that is totally negilgible when the amount of value
		expressions generated are of a small amount but that cost grows with amount of value expressions generated are of considerable
		amount, to divert the comptime cost to runtime cost and reduce the final binary cost, runtime delay initialization support for
		static arrays is needed.

		- Why only static arrays you might ask? Because it is the only inflexible data structure

		The LCI attribute `use_while_loop_for_delay_initialization_of` requires a while loop that must contain the following:
		- An index object qualified with the LCI attribute `index_object`.
		- The index object must be initialized with `1'ui` value.
		- A boolean expression that constraints the loop to the size of the category 2 type 1, exactly written like the following:
		`<index> lt or eq <size>`.
		- The <size> value must be gotten from the static array size field
		- A modifier expression that increments the index object by one.
		- A while body (i.e a block expression) that contains what is known as the crux expression that involves the initialization of the
		objects behind the static array with the following specifications:
			- The objects behind the static array can only be delay initialized, it is not allowed to be used
			after(and obviously before) it's initialization. This limitation rule is logically sound as any of use of the contents in
			the objects is either unsafe to use i.e if used before initialization or already available to use if coded correctly i.e
			if used after initialization.
			- The use of the `unsafe_nth` function marco to get the specific object behind the allocating MAR which can only be called once.
			Since the `unsafe_nth` function marco can only be called once, it is allowed to be mutably referenced in the case where the
			type of the objects behind the static array is also a static array and in need of
			delay initialization using the while loop also, but the mutable reference is still guided by the same rules of the crux
			expresion. Surprisinly, there are no specific lifetime restrictions on the mutably reference like mandating that it's
			lifetime equals that of the while loop's block lifetime(essentially allocated along with the block), this is
			because after initialization, although it cannot be used as per the crux expression specification, it is still very much
			valid and safe outside the while loop.
			- The use of the index object as the value expression argument to the `unsafe_nth` function marco
			- The use of the delay initialization operator to assign an initial value to the specific object behind an alocating MAR.
			- Any number of valid block expression items are allowed to be defined before and after the crux expression.


		Example:
		```(nc-lang)
		type [20]ui4 odds

		#at:use_while_loop_for_delay_initialization_of(odds)
		while #at:index_object infer i = 1'ui; i lt or eq odds.size; i += 1 {
			odds.!unsafe_nth(i) := i as ui4 + 2
		}

		type [10][12]ui4 x

		#at:use_while_loop_for_delay_initialization_of(x)
		while #at:index_object infer i = 1'ui; i lt or eq x.size; i += 1 {
			infer x_ith = &mut x.!unsafe_nth(i)

			#at:use_while_loop_for_delay_initialization_of(x_ith*)
			while #at:index_object infer j = 1'ui; j lt or eq x_ith*.size; j += 1 {
				x_ith*.!unsafe_nth(j) := j as ui4
			}
		}

		type [90][10][10]ui1 y

		#at:use_while_loop_for_delay_initialization_of(y)
		while #at:index_object infer i = 1'ui; i lt or eq y.size; i += 1 {
			infer y_ith = &mut x.!unsafe_nth(i)

			#at:use_while_loop_for_delay_initialization_of(y_ith*)
			while #at:index_object infer j = 1'ui; j lt or eq y_ith.size; j += 1 {
				infer y_ith_ith = y_ith*.!unsafe_nth(j)

				#at:use_while_loop_for_delay_initialization_of(y_ith_ith*)
				while #at:index_object infer k = 1'ui; k lt or eq y_ith_ith.size; k += 1 {
					y_ith_ith*.!unsafe_nth(k) := k as ui4
				}
			}
		}

		type [10][12]ui4 z

		#at:use_while_loop_for_delay_initialization_of(z)
		while #at:index_object infer i = 1'ui; i lt or eq z.size; i += 1 {
			z.!unsafe_nth(i) := [for item in 1~10 {item}] /# The inner category 2 type 1 decides to opt out of while delay initialization
		}

		std:io:!printf("%{odds}=, %{x}=, %{y}= and %{z}=")
		```

		GROUPED USE OF STATIC ARRAY OBJECTS DURING INITIALIZATION
		Imangine this example:
		```(nc-lang)
		type [1024][1024]r4 (a, b, c)
		type random rand()

		#at:use_while_loop_for_delay_initialization_of(a)
		while #at:index_object infer i = 1'ui; i lt or eq a.size; i += 1 {
			infer a_ith = &mut a.!unsafe_nth(i)

			#at:use_while_loop_for_delay_initialization_of(a_ith*)
			while #at:index_object infer j = 1'ui; j lt or eq a_ith*.size; j += 1 {
				a_ith*.!unsafe_nth(j) := rand.next()
			}
		}
		#at:use_while_loop_for_delay_initialization_of(b)
		while #at:index_object infer i = 1'ui; i lt or eq b.size; i += 1 {
			infer b_ith = &mut b.!unsafe_nth(i)

			#at:use_while_loop_for_delay_initialization_of(b_ith*)
			while #at:index_object infer j = 1'ui; j lt or eq b_ith*.size; j += 1 {
				b_ith*.!unsafe_nth(j) := rand.next()
			}
		}
		#at:use_while_loop_for_delay_initialization_of(c)
		while #at:index_object infer i = 1'ui; i lt or eq c.size; i += 1 {
			infer c_ith = &mut c.!unsafe_nth(i)

			#at:use_while_loop_for_delay_initialization_of(c_ith*)
			while #at:index_object infer j = 1'ui; j lt or eq c_ith*.size; j += 1 {
				c_ith*.!unsafe_nth(j) := 0
			}
		}
		```

		The example above is legal nc-lang code but there is a huge problem, why should it take three loops to initialize `a`, `b` and
		`c` when they all have the same size. Seperating the initialization to three distinct loops when they all the same size lacks
		machine simpathy, to fix this the language allows grouping of while loop delay initialization of static arrays provided they have
		the same sizes. In this grouped initialization the following holds true:
		- Any of the sizes from the category 2 type 1 objects can be used as the size object
		- The single index object defined is used for all the category 2 type 1 allocating MAR participating in the group initialization

		The imagine example modified:
		```(nc-lang)
		type [1024][1024]r4 (a, b, c)
		type random rand()

		#at:use_while_loop_for_delay_initialization_of(a, b, c)
		while #at:index_object infer i = 1'ui; i lt or eq a.size; i += 1 {
			infer (a_ith, b_ith, c_ith) = &mut (a, b, c).!unsafe_nth(i)

			#at:use_while_loop_for_delay_initialization_of(a_ith*, b_ith*, c_ith*)
			while #at:index_object infer j = 1'ui; j lt or eq b_ith*.size; j += 1 { /"b_ith*.size used as the size object here, makes no difference if it were c_ith*.size or a_ith*.size"/
				(a_ith*, b_ith*, c_ith*).!unsafe_nth(j) := (rand.(next(), next()), 0)
			}
		}
		```

		LOOP UNROLLING
		As the while loop used for delay initialization is strictly defined by the language and the size of the dataset is known at
		comptime, it is the perfect candidate for compiler supported loop unrolling, which can add performance benefits to your code by
		operation addition optimizations if used correctly.

		```(nc-lang)
		type [20]ui4 odds

		#at:use_while_loop_for_delay_initialization_of(odds), #at:unroll_loop(5)
		while #at:index_object infer i = 1'ui; i lt or eq odds.size; i += 1 {
			odds.!unsafe_nth(i) := i as ui4 + 2
		}

		/# The loop now under 4 iterations instead of 20 iterations effectively expanding to the below:
		/"
			while infer (i, (t2, t3, t4, t5)) = (1'ui, 0'ui); i lt or eq odds.size; i += 4 {
				(t2, t3, t4, t5) = i + (1, 2, 3, 4)
				odds.!unsafe_nth(i) := i as ui4 + 2
				odds.!unsafe_nth(t2) := t2 as ui4 + 2
				odds.!unsafe_nth(t3) := t3 as ui4 + 2
				odds.!unsafe_nth(t4) := t4 as ui4 + 2
				odds.!unsafe_nth(t5) := t5 as ui4 + 2
			}
		"/
		```
	}

	INITIALIZATION:
	From the definitions of nc-lang objects and non-objects, one can tell that the only difference between them is the value
	contained, one's value is valid for the object's type and the other is not. The reason behind such assertion is because
	an object is given an initial value upon allocation while a non-object is not. But how can nc-lang make such a bold assertion
	of non-objects when in the realm of computers main memory is volatile so a non-object might get valid values of the it's type.
	Yes, the following is fact, but the operative fact being "main memory is volatile", while a value might be valid for a type, it
	does not guarantee it would be valid for another type and it also enters the realm of unpredicitability because the value of the
	non-object is guaranteed to change in every run of the program. So to prevent such, it is a law in nc-lang that all user allocated
	memory regions must be given an initial value. Although strict in it's nature, nc-lang makes the implementation of said law
	flexible with delay initialization.

	The process wherby a non-object gets an initial value is called initialization.

	VALUE SEMANTICS defines what exactly are values in nc-lang, how they are treated and passed around.

	A VALUE in nc-lang is just simply data that has been made meaningful due to a type.

	TYPES OF VALUES
	- Concrete values
	- Abstract values

	Concrete values have an identity and mutability
	Abstract values have no identity and mutability

	VALUE PASSING SEMANTICS

	ASSIGNMENT OPERATION SEMANTICS
	The semantics of an assignment operation is the basis that all values use to be passed around. Meaning they are used in function
	arguments entry. The operation is a non-value expression operation.

	VARIOUS CASES OF VALUE PASSING:
	- Assignment operation
	- Initial assignment operation
	- Delay initial assignment operation
	- Function argument entry

	<expression>₁ - refers to the lhs operand of an assignment operation
	<expression>₂ - refers to the rhs operand of an assignment operation

	The lhs operand requires that the rhs operand be of the same type and nothing more. The language does not support any forms
	of implicit type conversions or cocercion during assignment. This is where the strong typing of nc-lang lies, the language
	refuses to make exceptions even though the types can be interchanged or are synonymous like integers which an example would
	be given later on.

	Types Of Value Passing
	- Copy value passing: an rhs-operand is copied to an lhs-operand, leaving the rhs-operand unchanged.
	- Move value passing: an rhs-operand is copied to the lhs-operand but the lhs-operand loses it's contents. The lhs-operand
	  loses it's content by implicitly getting defaulted by the compiler.

	How values are passed is dependant on the type:
	- The values of primitive types except allocating MARs follow copy value semantics
	- The values of other types follow move value semantics by default, but can choose copy semantics only if it is eligible.

	CLONE
	Values that follow move value semantics can use the clone contract to clone new values that can be later moved.

	DROP SEMANTICS
	Values using copy value semantics cannot be dropped, effectively forcing types that manages resources to use move value semantics
	if they want automatic cleanup before an assignment operation and before lifetime end. A drop function can only be called in
	another drop function.

	Before an assignment operation, the compiler applies the drop function on the lhs-operand

	Example of a move value semantic applied on an assignment operation
	```(nc-lang)
	fn main {
		type array x = array[1, 2, 3, 4]
		type array y = array[3, 7, 3]
		/# x.drop() -- gets cleaned up before reassignment because it uses move value semantics
		x = array[3, 4, 5]
	} /# x.drop() -- gets cleaned up at the end of it's lifetime because it uses move value semantics
	```

: Circular dependencies are evil (Global object initialization, type creators, MARs)
{
	- Circular dependency on global object initialization
	{
		Internal global object initialization order is defined by the language but external global objects initialization order is
		not defined by the language, for this reason, internal global objects cannot depend on external global object or any function or marco that makes use of them
		for it's initialization.

		- Scope visibility handles the cases where internal global object initialization depends on uninitialized internal global
		  objects
		- Circular imports are supported under this semantic

		```(nc-lang)
		/$ a.ncl
			import mod b
			struct a = type si4 (num, num2)

			scope @a {
				fn new(type si4 num2) { type a(b:b.num, num2) }
			}
			#at:thread_local
			type a a = @a:new(45'si4) /$ ERROR: The initialization of internal global objects cannot depend on external global objects

		/$ b.ncl
			import mod a
			struct b = type si4 (num, num2)

			scope @b {
				fn new(type si4 num2) { type b(a:a.num, num2) }
			}
			#at:thread_local
			type b b = @b:new(67'si4) /$ ERROR: The initialization of internal global objects cannot depend on external global objects
		```
	}
	- Circular dependency on struct, union and unique type creators
	{
		The size of a type must be known at comptime, therefore features such as circular dependencies between types that prohibit
		this rule are forbidden, use of a reference is preferred or better yet an allocating MAR to own the data

		```(nc-lang)
		struct xT = type yT y /$ ERROR: circular dependency detected
		struct yT = type xT x /$ ERROR: circular dependency detected
		```
	}
	- Circular dependency on memory address references
	{
		```(nc-lang)
		struct node = type *node node
		type *node mut x()
		type *node y(x)
		node = y
		```

		Due to allocating MARs semantics, circular dependencies are impossible to be represented, only non-allocating MARs unwillingly
		support circular dependencies.
	}

: When checking for constraints by the compiler (EHMM?)
{
	- To enforce a constraint where a bad actor is constrained by a runtime conditional, the compiler must
	  behave as though the conditional is a true case, meaning it must behave as though the bad actor is going
	  to be executed.

	- To enforce a constraint where a bad actor's behaviour is overriden by a good actor but the good actor is
	  constrained by a runtime conditional, the compiler must behave as though the conditonal is a false case
	  meaning it must behave as though the good actor would not get executed.

{

	lifetime(lifetime that persists throughout the entire execution of a program).

	- Number literals
	{

		- Real number literals

		  - IEEE-754 binary floating point are stored as is whithout conversion from the frontend, LLVM handles that.

: Object construction correctness
{
	Access restriction and factory functions alone cannot be used to ensure object construction correctness,
	there needs to be LCI attributes that ensure object construction correctness.

	The current way object construction correctness works and way access restriction is not fit for it or even
	sound semantically for it.

	[CURRENT WAY]
	A type-creator, it's access-restriction semantics and factory functions are what controls object construction
	correctness, this is so because unlike in C++, I don't expose the actual object constructor to be redefined
	by the user, therefore limiting my only options to exposing a pre-defined unchangable primary object
	constructor to the user with access-restriction only hindering items of either a different module or
	package and items external-to-the-scope-of-the-type from accessing the primary constructor and creating a
	dependence on factory functions alone to attempt to enforce object construction correctness. Because of
	this, the model is insanely unacceptable for types that need to maintain an invariant throughout it's
	visible lifetime, that is from it's construction till when it is dropped, because even with the combination
	of access-restriction and factory functions, the primary constructor is still left exposed to certain items
	depending on the access-restriction used and those items can intentionally or worse unintentionally use the
	primary constructor thereby bypassing the factory function that enforces the invariant of the type.
	Examples in both C++ and ncLang:
	```(c++)
	class IntegerEndExclusiveRange {
		std::unit32_t m_start;
		std::unit32_t m_end;

		IntegerEndExclusiveRange(std::unit32_t start, std::unit32_t end) {
			if (start > end) {
				std::err << std::format("The start({}) cannot be greater than the end({})\n", start, end);
				assert(false);
			}

			return IntegerEndExclusiveRange{start, end}
		}

		void create_and_print_range(std:uint32_t start, std::unit32_t end) {
			auto range = IntegerEndExclusiveRange{start, end} // The object constructor has be defined to enforce the invariant
			std::cout << "start{"
			for (int i = range.start; i < range.end; i += 1) {
				std::cout << i << '\n';
			}
			std::cout << "}end"
		}
	};
	```

	```(ncLang)
	struct integerEndExclusiveRange = 'at:hide obj (start, end): ui4

	scope @integerEndExclusiveRange =
	fn new(obj (start, end): ui4) {
		if start gt end {
			panic(format[string]("The start start({start}) cannot be greater than the end({end}\n)"))
		}

		-> obj: integerEndExclusiveRange(start, end)
	}

	fn create_and_print_range(obj (start, end): ui4) {
		obj range = obj: integerEndExclusiveRange(start, end) >! The object constructor does not enforce the invariant, but the factory function new does, but there is nothing stopping this function from using it
		printf("start\{Left Curly Bracket}")
		while obj i = start; i lt end; i += 1 {
			printf("{i}\n")
		}
		printf("\{Right Curly Bracket}end")
	}
	end @integerEndExclusiveRange scope
	```

	With the C++'s model, the invariant of `start` always being less-than `end` is maintained because the
	primary constructor has been defined to enforce the invariant regardless of if the variables(objects) are
	`private`, but in ncLang, with the access-restriction LCI attribute preventing items external to it's module
	from accessing them, items in the module can still access the primary constructor which does not enforce
	the invariant and when the access-restriction LCI attribute is changed to `'at:hide(type)`, other items in
	the scope of the type can still access the primary constructor which does not enforce the invariant and so
	on.

	ACCESS RESTRICTION EVEN BEING AMONG THE ENFORCERS IS INHERENTLY FLAWED
	Access-restriction in it's entireity is meant to solely manage the access of an object in a type-creator
	(field) after it's construction, not as it is being constructed. Example, when I make a field access
	restricted to it's module, the primary object constructor should still be available to me because, it
	simply does not equate access the field direcly, only the direct access of the field in a different module
	should warrant compliant from the compiler.

	[New Way]
	Due to the flaws of the previous system, I introduce a new model that takes into account object constructor
	correctness and removes access-restriction semantics thereby making it semantically sound. The introduction
	of both LCI attributes `'at:restrict_primary_object_constructor` and `'at:factory` help manage this.
	- `'at:restrict_primary_object_constructor` can be used to qualify any type-creator that wishes to restrict it's
	object constructor to factories.
	- `'at:factory` can be used to tell the compiler that a function or marco is a factory. Factories are
	function or marcos scoped to a type that are allowed to use the object constructor of a type when it is
	restricted by the type for all other items but must compley with constraint of it's give-type being either
	the type itself or failure handling type whoose good state is the type itself.

	```(ncLang)
	'at:restrict_primary_object_constructor
	struct integerEndExclusiveRange = 'at:hide obj (start, end): ui4

	scope @integerEndExclusiveRange =
	'at:factory
	fn new(obj (start, end): ui4) {
		if start gt end {
			panic(format[string]("The start start({start}) cannot be greater than the end({end}\n)"))
		}

		-> obj: integerEndExclusiveRange(start, end)
	}

	>! ERROR this function is not a factory
	fn create_and_print_range(obj (start, end): ui4) {
		obj range = obj: integerEndExclusiveRange(start, end) >! The object constructor does not enforce the invariant, but the factory function new does, but there is nothing stopping this function from using it
		printf("start\{Left Curly Bracket}")
		while obj i = start; i lt end; i += 1 {
			printf("{i}\n")
		}
		printf("\{Right Curly Bracket}end")
	}
	end @integerEndExclusiveRange scope
	```

	[Default object constructor]
	Default object constructors can access the primary constructor at will.
	```(nc-lang)
	struct x = type (si4, &str) (a, b)

	scope @x =
	marco 'default() {
		obj: x(30, "Daniel")
	}
	end @x scope
	```

	Note that, `'at:restrict_primary_object_constructor` does not affect default object constructors.
